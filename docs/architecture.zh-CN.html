Concepts / Architecture

核心 runtime 和基础设施分层维护。

localmelo 的核心约束是：melo/ 负责 agent runtime 与 contracts， support/ 负责 config、backend、provider、onboarding 和 gateway。 这样 agent loop、memory、checker、executor 可以独立于具体模型服务演进。

运行时路径

Agent 规划

Agent 读取 working memory 和 long-term retrieval，解析工具提示，并调用 chat provider 生成下一步。

Checker 校验

Gateway ingress、planning、executor request、executor result 和 memory write 都通过 checker 边界。

Executor 执行

Executor 只负责结构化工具执行：registry lookup、workspace policy、timeout、artifact metadata。

Memory 分层

Working memory、history、long-term memory 和 tool registry 是不同层，不混成一个隐式 context。

需要保持的边界

边界规则原因
melo/ 和 support/melo/ 不直接 import infrastructure implementation。核心 runtime 保持可移植、可测试。
Agent 到 ExecutorTool call 先经过 checker 和 workspace policy。危险 shell / file 操作在执行前被拒绝。
Agent 到 MemoryMemory write summary 先经过 checker，再进入长期存储。控制 prompt 和 memory 增长。
Backend本地 backend 只做 connector。模型 runtime 的启动和管理不属于 localmelo core。

实现状态

进行中

Memory system

计划中

Sleep pipeline

Training / evaluation 目前仍是 scaffold。下一步应该先写 design issue，而不是直接实现。
